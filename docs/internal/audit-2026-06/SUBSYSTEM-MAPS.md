# Subsystem maps (from the 8 audit agents)

Reference for executors: how each subsystem actually works and what is/isn't on the hot path.

## hot-path-protocol

The hot path lives almost entirely in src/server.rs: a single accept loop (ProxyServer::run, line 501) spawns one tokio task per client connection into handle_client -> client_loop, which frames client bytes with ProtocolCodec (src/protocol.rs, tag+length framing into BytesMut with O(1) split_to), runs optional anomaly/plugin hooks, then calls route_and_forward per message. route_and_forward classifies the SQL verb (is_write_query, uppercase prefix match), picks a node from RwLock-guarded health/round-robin state, dials a raw TcpStream to the backend when switching (no pooling, no TLS on this path), writes the re-encoded frame, and store-and-forwards the entire backend response into a Vec until it sees ReadyForQuery before writing anything back to the client. src/backend/ (client.rs + stream.rs/tls.rs/auth.rs) is a separate hand-rolled, text-format-only PG client used solely for internal management queries (health, failover replay), not for relaying client traffic; it supports TLS+SCRAM and shares the protocol.rs codec. The architecture is strictly one-message-at-a-time request/response: no pipelining, no extended-protocol awareness (Parse/Bind/Execute each wait for a ReadyForQuery that never comes until Sync, hitting a 30s timeout), and the pool-modes ConnectionPoolManager is constructed but never consulted on the data path. The dominant per-query costs found are quadratic buffer cloning in the response relay, full-response buffering, per-read Vec allocations, up to ~7 redundant clones/parses of each Query payload across hook sites, missing TCP_NODELAY everywhere on the data path, and async-lock acquisitions (including a global write lock for the round-robin counter) on every routed query.

## pooling-balancing

The pooling-balancing subsystem has three layers. src/connection_pool.rs is the base per-node pool: a single Arc<RwLock<HashMap<NodeId, NodePool>>> holding idle PooledConnections (each carrying an OwnedSemaphorePermit and optional hand-rolled BackendClient), with acquire popping an idle entry or creating a new connection under a semaphore-bounded permit, and a 30s server task driving evict_idle. src/pool/ adds mode awareness on top: ConnectionPoolManager (DashMap of node pools + active leases) hands out ConnectionLease objects whose on_statement_complete parses SQL via TransactionEvent::detect to decide Hold/Reset per session/transaction/statement mode, with release running a DISCARD ALL reset round trip for non-session modes; hardening.rs adds leak/stale/exhaustion bookkeeping behind parking_lot locks. src/load_balancer.rs is a strategy-based node selector (round-robin/weighted/least-conn/latency) over an async RwLock map, exposed through sync methods that call Handle::block_on. src/multi_tenancy/pool.rs is counters-only accounting (atomics per TenantPool behind a DashMap keyed by String TenantId) with no real connections. Critically, tracing the per-query path in src/server.rs shows none of this is on the hot path today: route_and_forward opens raw TcpStreams and re-authenticates on every backend switch, reimplements read/write selection with its own full-string to_uppercase scans, and never calls ConnectionPoolManager::acquire or LoadBalancer::select_for_*, so the subsystem's cost today is mostly latent — but it becomes the per-query critical section (global write locks, dashmap-guard-across-await, per-release DISCARD ALL) the moment it is wired in.

## routing-rewrite-analytics

The subsystem is five feature-gated per-query layers: src/routing/ (HintParser with two LazyLock regexes -> QueryRouter holding nodes in a tokio RwLock<Vec<NodeInfo>> -> NodeFilter which builds Vec<String> 'reasons' via format! per query -> RoutingMetrics); src/rewriter/ (SqlParser string-scan parse -> RuleMatcher fingerprint/regex/table indexes -> TransformationEngine string surgery, rules behind parking_lot RwLock); src/analytics/ (QueryAnalytics::record fans one QueryExecution — itself ~7 owned Strings — into fingerprinter (6 chained regexes), StatisticsStore (DashMap<u64,QueryStatistics> with scan-based eviction), SlowQueryLog (sync file write), PatternDetector (per-session VecDeque history + window scans), intent classifier, WorkflowTracer, CostAttribution); src/schema_routing/ (QueryAnalyzer ~30 contains() scans over an uppercased copy, SchemaRegistry DashMaps that deep-clone TableSchema per lookup, SchemaAwareRouter cloning node vectors per decision, SchemaRoutingMetrics with global tokio RwLock<HashMap> writes per query); and src/lag/ (LagMonitor + DashMap RYW tracker; LagAwareRouter allocates format! reason Strings inside every decision). Notably, grep shows none of these modules are referenced from server.rs's route_and_forward hot loop (which uses its own anomaly_fingerprint and plugin route hook), so these costs apply when the corresponding features are wired/enabled — but when they are, every Query message pays the full stack: hint regex scan, 1-2 write-detection case conversions, fingerprint normalization, intent classification (uppercase+lowercase copies), multiple DashMap entry insertions with cloned String keys, and 2-3 global lock acquisitions for metrics rings. The dominant systemic patterns are repeated full-query case-converted copies at each layer (no shared classification result), regex work that is either re-executed or re-compiled per query, deep clones of node/schema metadata per routing decision, and metrics bookkeeping that funnels all sessions through single locks or O(n) eviction scans.

## ha-failover-replay

The ha-failover-replay subsystem spans the control plane (src/failover_controller.rs orchestrating candidate selection, pg_promote and LSN catch-up; src/primary_tracker.rs polling pg_is_in_recovery per node to publish the current primary via parking_lot RwLock + broadcast) and the replay plane (src/transaction_journal.rs as the in-memory per-transaction statement journal behind one global tokio RwLock; src/failover_replay.rs replaying journaled statements onto a promoted node; src/replay/mod.rs reusing the same journal for admin time-travel replay; src/session_migrate.rs and src/cursor_restore.rs regenerating SET/PREPARE/DECLARE state on the new node; src/upgrade_orchestrator/ driving pub/sub-based version upgrades gated by src/switchover_buffer.rs). The steady-state hot path that HA actually touches lives in src/server.rs route_and_forward: per message it classifies write-vs-read by uppercasing the SQL, reads the shared tokio-RwLock health map (written by the serial TCP-connect health checker and cloned at 1 Hz by the admin sync task), takes a global lb_state write lock for round-robin, and on primary loss busy-polls at 500 ms inside select_primary_with_timeout. Critically, the journal and switchover buffer are constructed in ServerState but never fed from the data path — no begin_transaction/log_statement/buffer_query callers exist outside tests — so today's per-statement journaling overhead is zero, at the price of failover replay, time-travel replay, and cutover write-pausing all being inert. The remaining real per-query HA costs are the shared health/lb locks, two full-SQL allocations for routing classification, a per-message full-buffer clone in the backend response loop, and missing TCP_NODELAY; failover-time costs are dominated by serial probing, 500 ms wake polling, per-statement backend reconnects in replay/migration, and a replay-wide write lock held across awaits. data/wal exists but is empty — no durable journaling; everything is in-memory and lost on proxy restart.

## caching-tiers

The caching subsystem has three parallel, largely unintegrated stacks. (1) src/cache (feature query-cache) is the per-query result cache: QueryCache::get/put drives a per-connection L1 exact-match cache (l1_hot.rs, parking_lot RwLock + Vec-based LRU), a shared L2 keyed by a regex-normalized fingerprint (l2_warm.rs, DashMap plus an optional 'mmap' file store that actually does open/seek/read per hit and never reclaims space), and an L3 semantic tier (l3_semantic.rs) that calls an external Ollama embedding HTTP service and brute-force scans all entries with cosine similarity; invalidation.rs maintains table→key reverse indexes fed by WAL events. (2) src/distribcache (feature distribcache) is a second multi-tier design — HotCache (DashMap, Vec<u8> payloads cloned per hit), WarmCache (an 'SSD' tier that is really an in-RAM DashMap with synchronous zstd/bincode and a no-op LZ4 mode), DistributedCache (consistent-hash mesh that opens a fresh TCP connection per get/insert/invalidate), plus a workload classifier, heatmap analytics (global bucket write lock on every access), prefetcher, and an 852-line WAL invalidator that performs blocking std::net reads inside async fns while holding a tokio RwLock. (3) src/edge (feature edge-proxy, wired into server.rs) is the geo edge cache whose get() deep-copies the cached wire response and serializes all hits on a single write-locked O(n) VecDeque LRU, with home→edge invalidation fan-out over per-edge mpsc/SSE channels in registry.rs. The dominant per-query costs observed are the 6-9-regex normalization pipeline executed twice per cached query, per-hit write locks + O(n) LRU scans + payload deep-copies in every L1/edge tier, and connection-per-operation networking in the distributed tier; the declared request-coalescing (pending_requests) is dead code, so hot-key expiry stampedes the backend.

## plugins-wasm

The plugins-wasm subsystem implements a wasmtime-backed extension layer: src/plugins/loader.rs handles disk loading (bare .wasm + sidecar manifest, or packed .tar.gz with SHA-256 and Ed25519 trust-root verification); src/plugins/runtime.rs owns the shared wasmtime Engine, compiles modules, and executes hooks via call_hook using a JSON-in/JSON-out (ptr,len) ABI over guest linear memory; src/plugins/mod.rs provides PluginManager (DashMap of LoadedPlugin, RwLock<HashMap<HookType, Vec<String>>> hook registry) which fans out execute_pre_query/execute_route/execute_post_query/execute_authenticate. Supporting files: host_imports.rs (kv_get/set/delete + sha256_hex host functions over a global RwLock-ed KV map), metrics.rs (globally locked stats), hot_reload.rs (mtime poller whose check_updates has no callers outside the module). On the per-query hot path, server.rs invokes three hook stages per simple-Query message — apply_pre_query_hook (server.rs:789), apply_route_hook (server.rs:1137), fire_post_query_hook (server.rs:847) — each independently cloning the payload, parsing the SQL, and building a QueryContext (two SQL String copies + a UUIDv4) even when zero plugins are loaded, as long as plugins are enabled in config. When a plugin is loaded, each hook call synchronously builds a fresh Store and Linker, re-registers four host imports, re-instantiates the module, JSON-serializes the context per plugin, copies bytes into and out of linear memory, and records metrics through two global write locks — all on the tokio reactor thread with the epoch deadline disabled (u64::MAX) and the configured 100ms timeout unenforced. Extended-protocol (prepared-statement) traffic bypasses the hooks entirely, which is both a policy-coverage gap and the subsystem's main architectural ceiling.

## tenancy-auth-guards

The tenancy-auth-guards subsystem is the per-request policy layer of HeliosProxy: src/multi_tenancy/ resolves a TenantId from connection context (identifier.rs), routes by isolation strategy, and for row-level isolation rewrites SQL per query via transformer.rs (WHERE-clause injection driven by string find/uppercase, no SQL parser), recording per-tenant atomics in metrics.rs; src/auth/ provides JWT/OAuth/API-key/basic verification (handler.rs orchestrates providers behind a 60s RwLock<HashMap> result cache; jwt.rs signature verification is currently a stub, so JWT cost is base64+serde_json parse per uncached token) plus session token lookup (session.rs, three RwLock<HashMap>s); src/rate_limit/ combines DashMap-keyed token buckets, per-event-timestamp sliding windows, and semaphore-style concurrency limits with a string-heuristic cost estimator; src/circuit_breaker/ is a mostly-atomic state machine (breaker.rs) whose allow_request/RequestGuard path is cheap; src/anomaly/ is the only one of these wired directly into the server.rs message loop today — record_anomaly_observation runs on every Query message and feeds rate-spike (EWMA z-score), novel-fingerprint, and SQL-injection detectors. The dominant hot-path problems are concentrated in (a) the anomaly detector, which takes two-to-three process-global write locks per query and recomputes O(window) statistics with a fresh Vec inside the lock, (b) the rate-limit sliding window, which preallocates 480KB per limiter key and ignores configured limits, and (c) the row-isolation transformer, which performs ~12 full-query case-conversion allocations per SELECT and deep-clones TenantConfig per query. Auth is per-connection rather than per-query in the wired path, but its API-key verification is O(n)-scan with per-entry rehashing and its caches/stores serialize on global locks with unbounded-growth edge cases. Circuit-breaker bookkeeping is well designed (atomics, bucketless counter capped at 1000 events) and is not a bottleneck; notably its BucketedCounter is exactly the O(1) structure the rate-limit and anomaly windows should be using.

## infra-build-observability

The infra/build/observability subsystem comprises the build manifest (Cargo.toml — no [profile.release] tuning at all, tokio with features=["full"], optional prometheus/opentelemetry behind an unused-by-default 'observability' feature), the hand-rolled admin HTTP server (src/admin.rs, 2257 LoC: one tokio task per connection, Connection: close, manual header parsing; serves /health, /metrics in JSON and hand-formatted Prometheus text, /topology, /api/sql TWR forwarding, chaos/replay/shadow/edge/plugin endpoints), and the wiring in src/server.rs where spawn_admin_server creates an AdminState and a 1s sync task that copies atomic ServerMetrics counters and the node-health HashMap into tokio-RwLock-guarded admin snapshots (so admin reads never touch hot-path locks — a good design). The per-query hot path (server.rs client_loop -> route_and_forward) carries the observability tax directly: every Query message is parsed three separate times with payload clones (anomaly observation, wasm pre-query hook, write classification), an RFC3339 timestamp string is formatted per query, and the anomaly detector (src/anomaly/) takes two global parking_lot write locks plus to_lowercase/to_uppercase full-string allocations per query, with unbounded fingerprint/rate-window maps. Independent of observability, route_and_forward has two every-query defects: an O(n^2) response_buffer.clone() per chunk while scanning for ReadyForQuery, full double-buffering of each backend response before forwarding, and no TCP_NODELAY on any data socket. The graphql/ gateway (engine.rs) is effectively inert for performance purposes — its SQL execution stage is explicitly mocked ("mock for now", engine.rs:433) and its metrics use std Mutex<HashMap> — so it adds binary size but no hot-path cost. benches/ (pooling.rs, routing.rs) exercise only object construction and feature-gated routing micro-ops; the actual proxy data path, codec, hook chain, and p99-under-contention behavior are entirely unmeasured.
