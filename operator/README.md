# `heliosdb-proxy-operator` — scaffold

Design + stub layout for T1.1, the Kubernetes operator + Helm chart.
The full operator lives in its own repo (`heliosdb-proxy-operator`);
this directory is the contract — CRD definitions + expected controller
behaviour — checked in alongside the proxy so the two evolve together.

The actual Go code, Helm chart manifests, and controller reconcilers are
generated by `operator-sdk init` in the separate repo. When that repo
lands, this directory should point to it.

## CRDs

### `HeliosProxy` (per-instance config)

```yaml
apiVersion: helios.heliosproxy.dev/v1alpha1
kind: HeliosProxy
metadata:
  name: analytics
spec:
  image: ghcr.io/dimensigon/hdb-heliosdb-proxy:0.4.0
  replicas: 2
  listenPort: 5432
  adminPort: 9090

  nodes:
    - host: pg-primary.db.svc
      port: 5432
      role: primary
      weight: 100
    - host: pg-standby.db.svc
      port: 5432
      role: standby
      weight: 100

  pool:
    minConnections: 5
    maxConnections: 100
    idleTimeoutSeconds: 300

  features:
    tr: true
    poolModes: true
    queryCache: false
    rateLimiting: true
    auditChain: false

  plugins:
    enabled: false
    pluginDir: /var/lib/helios/plugins
    hotReload: false

  poolProfileRef:
    name: default-pool
  routingRuleRefs: []
  auditPolicyRef: ~
  tenantQuotaRefs: []

status:
  phase: Ready | Pending | Degraded | Failed
  currentPrimary: pg-primary.db.svc:5432
  healthyNodes: 2
  unhealthyNodes: 0
  readyReplicas: 2
  lastFailover: 2026-04-24T12:34:56Z
  conditions:
    - type: Ready
      status: "True"
      reason: AllNodesHealthy
```

### `PoolProfile`

Named pool configuration referenceable from multiple `HeliosProxy`s.

```yaml
apiVersion: helios.heliosproxy.dev/v1alpha1
kind: PoolProfile
metadata:
  name: high-tps
spec:
  mode: transaction          # session | transaction | statement
  maxPoolSize: 200
  minIdle: 20
  idleTimeoutSeconds: 60
  maxLifetimeSeconds: 1800
  acquireTimeoutSeconds: 5
  preparedStatementMode: track
  resetQuery: DISCARD ALL
```

### `RoutingRule`

Routing hints in YAML (mirrors `/*helios:route=...*/` comment syntax).

```yaml
apiVersion: helios.heliosproxy.dev/v1alpha1
kind: RoutingRule
metadata:
  name: analytics-reads-to-standby
spec:
  match:
    applicationName: ["analytics-*"]
    queryPattern: "^SELECT .*FROM events"
  route: standby
  consistency: eventual
  maxLagMillis: 5000
```

### `AuditPolicy`

Ties the `audit-chain` plugin (T2.4-P2) to a deployment.

```yaml
apiVersion: helios.heliosproxy.dev/v1alpha1
kind: AuditPolicy
metadata:
  name: pci-audit
spec:
  backend:
    type: s3
    bucket: acme-pci-audit
    region: us-east-1
  hashChain: true
  retentionDays: 2555                # 7 years
  includedTables: ["payments", "cards"]
  excludedUsers: ["monitoring"]
```

### `TenantQuota`

Per-tenant resource limits consumed by the multi-tenancy subsystem.

```yaml
apiVersion: helios.heliosproxy.dev/v1alpha1
kind: TenantQuota
metadata:
  name: free-tier
spec:
  maxConcurrentConnections: 10
  maxQueriesPerMinute: 600
  maxBytesReadPerMinute: 100_000_000
  costBudgetDollarsPerDay: 5
```

## Controller reconcile semantics

Every reconcile pass:

1. Fetch the `HeliosProxy` and its referenced `PoolProfile` /
   `RoutingRule` / `AuditPolicy` / `TenantQuota` objects. Missing refs
   surface as `Condition: RefMissing`.
2. Template the proxy config (`proxy.toml` content) from these CRDs.
3. Update a `ConfigMap` named `<helios>-config` with the rendered TOML.
4. Ensure a `Deployment` named `<helios>` exists with
   `spec.replicas = .spec.replicas`, mounted `ConfigMap`, two ports
   exposed (listen + admin), and `readinessProbe: /health/ready`.
5. Ensure a `Service` named `<helios>` of type `ClusterIP` exposing
   port 5432 (listener) and a separate `<helios>-admin` service for
   port 9090.
6. Poll each pod's admin `/nodes` and populate `.status.currentPrimary`
   / `healthyNodes` / `unhealthyNodes`.
7. On a detected primary change, annotate the Deployment with the
   new primary for operator observability.

## Topology auto-discovery

When `.spec.nodes` is empty, the operator discovers backend nodes via
the `postgres-topology` provider baked into the proxy: it reads a
`PostgresCluster` CR (from the CrunchyData or Zalando operators) or
falls back to a manual `Endpoints` resource named `<helios>-backends`.
See the proxy's `src/primary_tracker.rs::PostgresTopologyProvider`
for the underlying mechanics.

## Helm chart

Auto-generated from the CRDs. Minimum installation:

```sh
helm repo add helios https://charts.dimensigon.com/
helm install analytics helios/heliosproxy \
  --set nodes[0].host=pg-primary.db.svc \
  --set nodes[0].role=primary
```

Values YAML maps 1:1 to the `HeliosProxy` spec.

## Acceptance (T1.1 exit)

- `helm install heliosproxy ./charts/heliosproxy` on fresh minikube
  produces a working 3-node routing proxy in < 60 s.
- `kubectl get heliosproxy analytics -o yaml` shows
  `status.phase = Ready` once node health is green.
- Rolling the proxy image tag triggers a controlled rollout with
  zero dropped connections (validated via the T0-IT2 harness inside
  a kind cluster).
